# Raw Sources

This directory stores immutable input material for the second brain.

The purpose of `raw/` is simple: keep source material separate from the wiki.
Files here are inputs, not polished knowledge pages.

## Directory Structure
```text
raw/
├── README.md
├── assets/
├── trash/
├── manual/
│   ├── podcast/
│   ├── web/
│   ├── wechat/
│   ├── youtube/
│   └── x.com/
├── feeds/
│   ├── podcast/
│   ├── web/
│   ├── wechat/
│   ├── youtube/
│   └── x.com/
```

## Top-Level Layout

### `manual/`

Hand-curated sources that were intentionally saved into the second brain.

Use this for material you explicitly decided to keep, for example:

- an article you manually clipped
- a podcast transcript you decided was worth preserving
- a tweet thread or single post you deliberately captured
- a markdown note dropped into the pipeline for later ingestion

`manual/` is higher-signal by default because inclusion already reflects human
judgment.

### `feeds/`

Machine-captured sources that arrived from subscriptions, feeds, or ongoing
collection pipelines.

Use this for material gathered automatically, for example:

- RSS articles
- podcast episodes from subscribed feeds
- recurring x.com list snapshots
- feed-like captures from monitored sources

`feeds/` is optimized for throughput and breadth. It may contain more noise,
duplication, and low-priority items than `manual/`.


### `assets/`

Local binary or attachment-style files referenced by raw sources.

Use this for:

- PDFs
- images
- downloaded attachments
- supplementary files that are awkward to inline into markdown

If a raw markdown source depends on an external file, store that file here and
reference it from the markdown source when helpful.

### `trash/`

Holding area for raw inputs you no longer want in active circulation but are
not ready to delete permanently.

Use this sparingly. `trash/` should not become a second inbox.

## Source Types

Both `manual/` and `feeds/` are organized by source type when useful.

### `podcast/`

Transcript-style markdown files for podcast episodes.

Typical file shape:

- one markdown file per episode
- filename usually includes source name, episode title, and capture date
- content is a raw or lightly cleaned transcript, not a wiki summary

### `web/`

Saved web articles or long-form pages captured into markdown.

Use this for:

- blog posts
- essays
- shareholder letters
- product announcements
- pages converted from HTML into a markdown snapshot

These files should preserve the source material as faithfully as practical.

### `wechat/`

Markdown exports of WeChat articles or notes derived from WeChat content.

Use this when the source originates from the WeChat ecosystem and should remain
distinguishable from general web captures.

### `youtube/`

Captured YouTube-derived material, such as transcripts, notes, or video-linked
markdown captures.

This source type currently exists only under `manual/`, reflecting the current
workflow that YouTube items are intentionally saved rather than batch-ingested.

### `x.com/`

Captured x.com / Twitter material.

This directory may contain:

- direct single-post captures
- per-feed folders
- twitter-list snapshots
- index files generated by the capture pipeline

The structure under `x.com/` can be more nested than other source-type folders
because feeds may be grouped by account, list, or capture mode.

## Boundary With `wiki/`

- `raw/` stores source inputs.
- `wiki/` stores distilled knowledge.
- `manual/` and `feeds/` preserve where a source came from.
- Source files may be noisy, redundant, incomplete, or lightly cleaned.
- Wiki pages may be reorganized, merged, summarized, and rewritten freely.

The LLM may read files here, but should not rewrite them in place.
