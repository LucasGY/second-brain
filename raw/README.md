# Raw Sources

This directory stores immutable input material for the second brain.

The purpose of `raw/` is simple: keep source material separate from the wiki.
Files here are inputs, not polished knowledge pages.

## Directory Structure

```text
raw/
├── README.md
├── assets/
├── inbox/
│   ├── podcast/
│   ├── web/
│   ├── wechat/
│   └── x.com/
└── processed/
```

### `inbox/`

Landing area for newly captured sources that have not yet been fully absorbed
into `wiki/`.

Typical contents:

- podcast episode transcripts
- saved web articles
- WeChat article exports
- x.com / twitter-list captures
- manually dropped markdown notes that should later be ingested

`inbox/` should be treated as append-only in normal operation. New sources are
added here first, then referenced by downstream wiki work.

### `inbox/podcast/`

Transcript-style markdown files generated from podcast RSS feeds.

Typical file shape:

- one markdown file per episode
- filename usually includes source name, episode title, and date
- content is a raw or lightly cleaned transcript, not a wiki summary

This directory is optimized for machine capture, not for hand-curated writing.

### `inbox/web/`

Saved web articles or long-form pages captured into markdown.

Use this for:

- blog posts
- essays
- shareholder letters
- product announcements
- pages converted from HTML into a markdown snapshot

These files should preserve the source material as faithfully as practical.

### `inbox/wechat/`

Markdown exports of WeChat articles or notes derived from WeChat content.

Use this when the source originates from the WeChat ecosystem and should remain
distinguishable from general web captures.

### `inbox/x.com/`

Captured x.com / Twitter material.

This directory may contain:

- direct single-post captures
- per-feed folders
- twitter-list snapshots
- index files generated by the capture pipeline

The structure under `x.com/` can be more nested than other inbox folders
because feeds may be grouped by account, list, or capture mode.

### `processed/`

Machine-readable bookkeeping and artifacts for sources that have already gone
through part of the pipeline.

Use this for:

- deduplication state
- transcription bookkeeping
- processed ID registries
- other small state files that help automation avoid reprocessing

`processed/` is not the wiki and not a human-facing archive. It is operational
state that supports ingestion.

### `assets/`

Local binary or attachment-style files referenced by raw sources.

Use this for:

- PDFs
- images
- downloaded attachments
- supplementary files that are awkward to inline into markdown

If a raw markdown source depends on an external file, store that file here and
reference it from the markdown source when helpful.

## Boundary With `wiki/`

- `raw/` stores source inputs.
- `wiki/` stores distilled knowledge.
- Source files may be noisy, redundant, incomplete, or lightly cleaned.
- Wiki pages may be reorganized, merged, summarized, and rewritten freely.

The LLM may read files here, but should not rewrite them in place.
